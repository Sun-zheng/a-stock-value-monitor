# A股全市场 Buffett-Munger 每日分析手册

## 1. 目标

本项目用于在每个 A 股交易日运行一次“前一交易日全量数据”的价值分析流程，输出：

- 全市场扫描结果
- Buffett + Munger 九维分析结论
- 正式推荐与观察股票
- 本地结构化历史
- 飞书逐股记录
- 邮件摘要交付

当前默认不是“只在主板里挑”，而是默认全市场 A 股推荐：

- 推荐范围：`all_a_share`
- 覆盖板块：主板、创业板、科创板、北交所
- 排除范围：ST、退市、非股票证券
- 正式推荐上限：1 只
- 观察股票上限：5 只

## 2. 数据准确性口径

### 2.1 时间口径

- 每天分析使用前一交易日完整数据
- 行情交易日与估值交易日必须一致
- 不允许把旧盘中缓存冒充前一交易日收盘

### 2.2 行情与估值

- `src/data_fetcher.py` 优先使用可信的前一交易日收盘口径
- `src/data_source_manager.py` 负责按目标交易日命中缓存
- `src/freshness.py` 负责校验缓存是否仍可作为“前一交易日准确数据”

### 2.3 人工核验点

每天至少核验以下字段：

1. `估值数据交易日` 是否等于目标前一交易日
2. `行情交易日` 是否等于目标前一交易日
3. `行情数据类型` 是否为 `previous_trading_day_close` 或 `previous_trading_day_close_cache`
4. `数据性质` 是否明确说明价格、市值和安全边际都使用前一交易日收盘口径

## 3. 推荐范围与基准

### 3.1 推荐范围

推荐范围由 `config/strategy.json` 中的 `recommendation_scope` 控制：

- `all_a_share`：境内全部上市 A 股
- `main_board`：沪深主板

当前默认值为 `all_a_share`。

### 3.2 同行与相对估值基准

无论推荐范围是否切回主板，行业同行、相对估值和市场错配判断都应优先使用境内全部上市 A 股基准。

基准要求：

- 覆盖主板、创业板、科创板、北交所
- 排除 ST、退市、非股票证券
- 行业样本至少 8 只
- 对 PE / PB / PS 做 5% 到 95% 极值清洗

## 4. 完整分析逻辑

### 4.1 股票池

原始股票池来自 `Tushare stock_basic`。进入推荐流程前先做证券范围过滤：

- 保留当前推荐范围内的股票
- 排除 ST
- 排除退市
- 排除基金、转债、REIT、B 股和其他非股票证券

### 4.2 全市场轻筛候选

项目先对全市场做轻筛，再进入财务深度校验。默认候选上限是 500 只，由 `candidate_limit` 控制。

轻筛袖口包括：

1. 低估袖口
   - `PE TTM <= 15` 或 `PB <= 1.5` 或 `PS <= 1.5`
   - 总市值至少 20 亿
2. 质量袖口
   - `0 < PE TTM <= 25`
   - `0 < PB <= 4`
   - 总市值至少 20 亿
3. 高股息袖口
   - 股息率 `>= 2%`
   - 总市值至少 20 亿
4. 行业内相对低估袖口
   - PE / PB / PS 至少一项低于行业中位数
   - 行业样本充足
5. 综合补位
   - 当前四类袖口不足 500 只时，再按综合低估排序补齐

### 4.3 500 只候选的全量财务检查

进入深度分析的 500 只候选全部拉取财务、利润、资产负债表、现金流和质量字段，不允许只抽样一部分。

关键覆盖要求：

- 财报覆盖率至少达到正式推荐要求
- 现金流覆盖率至少达到正式推荐要求
- 四类财务期间必须尽量对齐到同一最新报告期
- ROE 使用完整年度口径，不接受季度简单年化替代正式判断

### 4.4 一票否决

以下任一条件触发时，不得进入正式推荐：

- 三大报表与财务指标报告期不一致
- 关键财务或现金流字段缺失
- 扣非净利润非正
- 经营现金流非正
- 可靠 ROE 非正
- 资产负债率过高
- 特殊行业暂无专用正式推荐模型
- 估值模型只能用于观察，不可用于正式推荐

### 4.5 三类独立估值

系统同时计算三类独立估值家族：

1. 同行相对估值
   - 行业 PE 中位数 × 标准化利润
   - 行业 PB 中位数 × 净资产
   - 行业 PS 中位数 × 完整年度收入
   - 三者取中位数作为相对估值族结果
2. 盈利绝对估值
   - 标准化利润 × 12
3. 现金流绝对估值
   - 标准化自由现金流 × 12

系统同时输出：

- 保守合理市值：三类估值中的最小值
- 中性合理市值：三类估值的中位数
- 乐观合理市值：三类估值中的最大值

正式安全边际定义为：

- `(保守合理市值 / 当前总市值 - 1) * 100`

只有三类独立估值都有效时，才计算正式安全边际。

### 4.6 Buffett + Munger 九维评分

九维由 `config/strategy.json` 的 `weights` 和 `quality_gates` 控制，当前框架包括：

1. 生意质量与护城河
2. 管理层与资本配置
3. 盈利能力与韧性
4. 财务安全
5. 现金流质量
6. 十年成长跑道
7. 估值与安全边际
8. 股东回报
9. 芒格反向清单

当前代码中的核心代理指标包括：

- ROE、扣非 ROE、ROIC
- 毛利率、净利率
- 经营现金流 / 净利润
- 资产负债率
- 收入多年趋势、利润多年趋势
- 股息率
- 货币资金 / 有息负债
- 商誉 / 净资产
- 自由现金流年度样本数
- 安全边际

正式推荐不仅看综合分，还要逐维满足质量门槛。

### 4.7 正式推荐硬门槛

正式推荐必须同时满足：

- 综合评分 `>= formal_score`
- 安全边际 `>= formal_margin`
- 可靠估值覆盖率 `= 100`
- 财报覆盖率 `>= 70`
- 现金流完整
- 无一票否决原因
- 估值模型允许正式推荐
- 九维长期质量门槛全部达标
- 市场错配判断不能是“未发现明确市场错配”

并且：

- 最多只保留 1 只正式推荐
- 即使没有正式推荐，也不能为了凑数降低门槛

### 4.8 观察股票规则

观察股票必须满足：

- 不是正式推荐
- 没有一票否决
- 综合评分 `>= watch_score`

并且：

- 最多保留 5 只
- 不得用低质量股票硬凑 5 只
- 如果不足 5 只，报告中必须明确“不足原因”

### 4.9 缺失数据原则

缺失数据只能输出“数据不足”，不能：

- 推断为正面证据
- 补造分值
- 因为模型缺失而放宽正式推荐门槛

## 5. 每日结果与比较

### 5.1 本地输出

每天基础流水线完成后，系统会写出：

- `reports/YYYY-MM-DD_result.json`
- `reports/YYYY-MM-DD_scan_summary.json`
- `reports/YYYY-MM-DD_report_base.md`
- `reports/YYYY-MM-DD_report.md`
- `reports/YYYY-MM-DD_email.md`
- `reports/YYYY-MM-DD_delivery_validation.json`
- `data/analysis_history/YYYY-MM-DD_analysis.json`
- `data/analysis_history/latest_analysis.json`

### 5.2 每日变化比较

`src/daily_comparison.py` 至少比较：

- 正式推荐列表
- 观察池列表
- Top10 候选
- 关键覆盖率与候选数量
- 交易日推进
- 报告正文变化

连续两天观察池完全相同，不自动视为错误。必须结合以下信息一起判断：

- 交易日是否推进
- 估值与行情日期是否更新
- Top10 是否变化
- 覆盖率或漏斗数量是否变化
- 报告正文是否变化

## 6. 邮件与飞书交付规则

### 6.1 邮件内容

邮件不是原始 JSON 转发，而是分析摘要，至少包括：

- 执行过程摘要
- 推荐范围
- 交易日和数据时点
- 覆盖率与候选漏斗
- 每只正式推荐的关键分析
- 每只观察股票的关键分析

每只股票至少保留以下分析字段：

- 当前价格
- 当前市值、流通市值
- PE / PB / PS
- 股息率
- ROE / ROIC
- 经营现金流、自由现金流
- 综合评分
- 安全边际
- 市场错配
- 已计价预期
- 关键证据
- 失败清单 / 反证清单
- 十年持有结论
- 下一步观察点

### 6.2 飞书写入

- 正式推荐和每只观察股票分别写入一条记录
- 当天逐股交付使用幂等键控制，避免重复写入
- 邮件状态回写到飞书记录

### 6.3 数值格式

- 所有数值保留两位小数
- 金额类字段 `>= 10000` 且 `< 100000000` 显示为“万”
- 金额类字段 `>= 100000000` 显示为“亿”
- 百分比字段统一显示 `%`

## 8A. 2026-06-24 规则补充

### 8A.1 深度复核 500 只的补足逻辑

- 正式深度复核目标仍是 500 只。
- 系统不会只对轻筛前 500 只直接做财务拉取，而是先按 `candidate_limit + buffer` 预取更多轻筛候选。
- 当前默认缓冲区是 200，因此目标 500 只时，通常会先预取最多 700 只轻筛候选的财务数据。
- 进入正式条件检查前，优先保留“最近完整年度财务指标齐备、报表期间一致、对齐报告期存在”的候选。
- 若财务齐备样本仍不足 500 只，才按原始轻筛顺序回补剩余名额。
- 报告和校验输出中应同时保留 `估值轻筛预取数量`、`估值轻筛通过数量`、`正式条件检查数量`、`financial_prefetch_requested`、`financial_ready_candidates`、`financial_selected_for_review`，用于审计 500 只深度复核是否真的全量完成。

### 8A.2 外发内容不一致时的自动修复

- `python main.py --validate-delivery` 会对比 `reports/YYYY-MM-DD_report.md` 的当前哈希与 `runtime_state` 中飞书、邮件交付记录的 `content_hash`。
- 同一天重复执行 `python main.py --deliver-final-report` 时，如果哈希未变化，幂等机制会跳过重复发送。
- 如果本地终稿哈希已变化，系统会自动把对应交付记录从 `completed` 重开为 `sending`，然后重新发送飞书和邮件。
- 因此“本地终稿与外发版本不一致”的场景不需要人工确认，修复动作就是再次执行一次 `python main.py --deliver-final-report`，系统会只重发需要追平的渠道。
- `--validate-delivery` 只要发现任一渠道仍保留旧哈希，就会返回非零状态，直到外发记录与当前本地终稿完全一致。

### 7.5 建议执行顺序

```powershell
pytest -q
python main.py --data-freshness-check
python main.py --strategy-validation
python main.py --run-pipeline
python main.py --deliver-final-report
python main.py --validate-delivery
```

## 8. 运维注意事项

1. 如果当天不是 A 股交易日，基础流水线应直接停止，不做对外交付。
2. 如果基础流水线未成功，`--deliver-final-report` 不应成为替代扫描命令。
3. 同一天重复运行 `--deliver-final-report` 时，幂等机制可能跳过重发。
4. 如果当天重跑基础流水线后本地终稿发生变化，要额外比对：
   - `reports/YYYY-MM-DD_report.md` 的当前哈希
   - `runtime_state` 中当天已交付记录的 `content_hash`
5. 若两者不一致，说明本地终稿与当天对外版本已经脱钩，需要人工决定是否重新交付。
